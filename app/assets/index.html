Problem Definition

Unlike a transaction on a local database, a distributed transaction involves altering data on multiple databases. Consequently, distributed transaction processing is more complicated, because the database must coordinate the commitment or the abortion of the changes in a transaction as a self-contained unit. In other words, the entire transaction commits, or the entire transaction aborts.

Two Phase Commit Protocol

The database ensures the integrity of data in a distributed transaction using the two-phase commit mechanism. In the prepare phase, the coordinator in the transaction asks the subordinates to promise to commit or abort the transaction. If all subordinates are able to commit the transaction, the transaction stages in the commit phase, otherwise it stages in the abort phase. During the commit phase, the coordinator asks all subordinates to commit the transaction. During the abort phase, all subordinates are asked to abort. You can view an illustration of the protocol below.

Figure 1: Two phase commit protocol diagram, note that an * next to the record type means that the transaction state is force-written to a stable storage.

We assume that each transaction is processed by both of the state charts showed below. When the transaction starts, the coordinator sends prepare messages to all subordinates. The coordinator awaits all subordinates to respond with their votes.

If all subordinates responded with a yes vote, the coordinator sends commit messages to all subordinates. If any subordinate responded to the prepare message with a "no" message or if a timeout occurred, the coordinator sends abort messages to all subordinates. After sending the "commit" or "abort" messages, the coordinator awaits all subordinates to respond with an ack message.

If a timeout occurs while waiting, the coordinator retransmits the "commit" or "abort" message to the respective subordinate. The transaction ends when all subordinates responded with an ack message.

Formalisation of the problem
Let \footnotesize N = \{ 0, 1, ..., n \} denote the set of nodes, which includes the coordinator and all subordinates.
Let \footnotesize n_0 \in N denote the coordinator.
Let \footnotesize n_i\in N, \quad i \in N^{+} denote any subordinate.
Let \footnotesize T = \{ 0, ... , m \} denote the set of transactions.
Let \footnotesize t \in T denote a transaction.
Let \footnotesize S = \{ Commit, Abort \} denote the set of possible states. Note that the states used here are independent from the states used in the state charts above.
Let \footnotesize f : N \times T \rightarrow S denote the function which maps any node and transaction to a state.
In order to reach a consistent state, following condition must be satisfied:
\large f(n_i,t) = f(n_j,t), \quad \forall i,j
Consistent state during two phase commit

The coordinator \footnotesize n_0 knows in which state it is in case of a failure, since the two phase commit protocol dictates the coordinator to keep a log.

(Theorem 1) \footnotesize f(n_i,t)=Commit \Rightarrow f(n_0,t)=Commit, i \in N^{+} (If a subordinate is in commit state, the coordinator must also be in commit state)

Proof: In order for subordinate \footnotesize n_i to be in commit state, the coordinator \footnotesize n_0 must have entered the commit state before \footnotesize n_i and sent a commit message to \footnotesize n_i. It is impossible for the coordinator to send a commit message, if it is not in the commit state.

(Theorem 2) \footnotesize f(n_i,t)=Abort \Rightarrow f(n_0,t)=Abort, i \in N^{+} (If a subordinate is in abort state, the coordinator must also be in abort state)

Proof: (Analogus to 1) In order for subordinate \footnotesize n_i to be in abort state, the coordinator \footnotesize n_0 must have entered the abort state before \footnotesize n_i and sent an abort message to \footnotesize n_i. It is impossible for the coordinator to send an abort message, if it is not in the abort state.

(Theorem 3) \footnotesize f(n_0,t)=Commit \Rightarrow f(n_i,t)=Commit, i \in N^{+} (If the coordinator is in commit state, all subordinates must also be in commit state)

Proof: If the coordinator \footnotesize n_0 enters commit state, he must send commit messages to all subordinates \footnotesize n_i. All subordinates must respond to the commit message with an ack message. If the coordinator doesn't receive an ack message from a subordinate after a fixed timeout, the coordinator retransmitts the commit message. It is impossible for the coordinator to send any other message than the commit message in the commit phase. Eventually all subordinates will receive the commit message, thus every subordinate \footnotesize n_i must be in commit state.

(Theorem 4) \footnotesize f(n_0,t)=Abort \Rightarrow f(n_i,t)=Abort, i \in N^{+} (If the coordinator is in abort state, all subordinates must also be in abort state)

Proof: (Analogus to 3) If the coordinator \footnotesize n_0 enters abort state, he must send abort messages to all subordinates \footnotesize n_i. All subordinates must respond to the abort message with an ack message. If the coordinator doesn't receive an ack message from a subordinate after a fixed timeout, the coordinator retransmitts the abort message. It is impossible for the coordinator to send any other message than the abort message in the abort phase. Eventually all subordinates will receive the abort message, thus every subordinate \footnotesize n_i must be in abort state.

(Theorem 5) \footnotesize f(n_i,t)=Commit \Leftrightarrow f(n_j,t)=Commit (If one node is in commit state, then all other nodes must also be in commit state)

Proof: We consider 3 possible cases:

Case 1: \footnotesize i \ne j = 0 ( \footnotesize n_i is subordinate, \footnotesize n_j is coordinator)

As shown in (1), if \scriptsize f(n_i,t)=Commit \Rightarrow f(n_j,t)=Commit, and as shown in (3) if \scriptsize f(n_j,t)=Commit \Rightarrow f(n_i,t)=Commit, thus \scriptsize f(n_i,t)=Commit \Leftrightarrow f(n_j,t)=Commit, for \scriptsize i \ne j = 0.

Case 2: \footnotesize j \ne i = 0 ( \footnotesize n_i is coordinator, \footnotesize n_j is subordinate)

As shown in (1), if \scriptsize f(n_j, t)=Commit \Rightarrow f(n_i,t)=Commit, and as shown in (3) if \scriptsize f(n_i,t)=Commit \Rightarrow f(n_j, t)=Commit, thus \scriptsize f(n_j,t)=Commit \Leftrightarrow f(n_i,t)=Commit, for \scriptsize j \ne i = 0.

Case 3: \footnotesize i \ne j, \quad i,j \ne 0 (both \footnotesize n_i,n_j are subordinates)

As shown in (1), if \scriptsize f(n_i,t)=Commit \Rightarrow f(n_0, t), and as shown in (3) if \scriptsize f(n_0,t)=Commit \Rightarrow f(n_j,t). Intuitively this means that WLOG if subordinate \scriptsize n_i is in commit state, the coordinator also must be in commit state and thus subordinate \scriptsize n_j also must be in commit state. Thus \scriptsize f(n_j,t)=Commit \Leftrightarrow f(n_i,t)=Commit, for \scriptsize i \ne j, \space i,j \ne 0.

From cases 1,2,3 follows that \footnotesize f(n_i,t)=Commit \Leftrightarrow f(n_j,t)=Commit. Simply put, it means that if a node is in commit state, any other node also must be in commit state.

(Theorem 6) \footnotesize f(n_i,t)=Abort \Leftrightarrow f(n_j,t)=Abort (If one node is in abort state, then all other nodes must also be in abort state)

Proof: (Analogus to 5) We consider 3 possible cases:

Case 1: \footnotesize i \ne j = 0 ( \footnotesize n_i is subordinate, \footnotesize n_j is coordinator)

As shown in (2), if \scriptsize f(n_i,t)=Abort \Rightarrow f(n_j,t)=Abort, and as shown in (4) if \scriptsize f(n_j,t)=Abort \Rightarrow f(n_i,t)=Abort, thus \scriptsize f(n_i,t)=Abort \Leftrightarrow f(n_j,t)=Abort, for \scriptsize i \ne j = 0.

Case 2: \footnotesize j \ne i = 0 ( \footnotesize n_i is coordinator, \footnotesize n_j is subordinate)

As shown in (2), if \scriptsize f(n_j, t)=Abort \Rightarrow f(n_i,t)=Abort, and as shown in (4) if \scriptsize f(n_i,t)=Abort \Rightarrow f(n_j, t)=Abort, thus \scriptsize f(n_j,t)=Abort \Leftrightarrow f(n_i,t)=Abort, for \scriptsize j \ne i = 0.

Case 3: \footnotesize i \ne j, \quad i,j \ne 0 (both \footnotesize n_i,n_j are subordinates)

As shown in (2), if \scriptsize f(n_i,t)=Abort \Rightarrow f(n_0, t), and as shown in (4) if \scriptsize f(n_0,t)=Abort \Rightarrow f(n_j,t). Intuitively this means that WLOG if subordinate \scriptsize n_i is in abort state, the coordinator also must be in abort state and thus subordinate \scriptsize n_j also must be in abort state. Thus \scriptsize f(n_j,t)=Abort \Leftrightarrow f(n_i,t)=Abort, for \scriptsize i \ne j, \space i,j \ne 0.

From cases 1,2,3 follows that \footnotesize f(n_i,t)=Abort \Leftrightarrow f(n_j,t)=Abort. Simply put, it means that if a node is in abort state, any other node also must be in abort state.

(Theorem 7) \footnotesize f(n_i,t)=f(n_j,t) (all nodes have the same state, consistent state)

Proof: As shown in (5) and (6), if any node is in commit (abort) state, then any other node must be in commit (abort) state. Thus

\large f(n_i,t)=f(n_j,t),\quad \forall i,j
\square
Simulations

We have conducted a series of simulations on the two phase commit protocols in order to observe it's behaviour according to different variables. We made simuations over the commit phase of the protocol in order to observe the time needed for a consistent state. The simulations were made using 2pc-runtime-simulator and uzh-2pc-benchmarks. Both simulators produced similar results. For the results below, we only made use of the former simulator. In order to proceed with the simulations, we first identified the variables in the experiment. Below you will find an enumeration of the identified variables:

  1. \small t_{runtime}: Runtime (time needed in order to reach the consistent state)
  2. \small p_{error}: Error rate (propability of a subordinate failure during commit)
  3. \small n_{sub}: Number of subordinates
  4. \small n_{commits}: Number of commits
  5. \small n_{parallel}: Number of concurrent commits (parallel operations)
  6. \small t_{message}: Message delay
  7. \small t_{recovery}: Recovery delay (time needed for a subordinate to recover from failure)
  8. \small t_{timeout}: Coordinator timeout (interval after which the coordinator assumes the subordinate had a failure)
  9. \small t_{backoff}: Backoff time (time the coordinator waits until contacting the subordinate again)

Obviously, \footnotesize n_{runtime} is the dependent variable. Note that, every simulation has been run with 1000 samples. All graphs below are histograms over the samples. Every histogram is accompanied by \footnotesize \mu which denotes the arithmetic mean and \footnotesize \sigma which denotes the standard deviation.

For our first round of simulations, we decided to variate the error rate whilst fixing all other variables. Below you can find an enumaration of all values from the fixed variables:

Comments on Simulation

Like in every computer simulation, many factors caused noise in our simulation. We expected that the results of our simulations would vary from machine to machine. All results shown here, were produced by a Macbook Pro 2015.

Besides the running machine, we also expected that our random number generator, which was responsible for causing uniformly distributed failures, wouldn't be perfectly uniform. We decided to use a third-party, open source library which helped us produce sufficiently uniform distributed values. We made use of random-js, which generated uniformly distributed random values thus ensuring failures where uniformly distributed as shown in Figure 16. We generated \scriptsize 1'000'000 values in the \footnotesize [0, 1) interval.

Observations

From the simulations we can see some interesting patterns emerge. Most notably, we observe that the runtime follows a normal distribution. We believe that the runtime follows a normal distribution due to the Central Limit Theorem. The CLT states that if a sample is obtained containing a large number of observations, each observation being randomly generated in a way that does not depend on the values of the other observations, and this procedure is performed many times, the computed values of the arithmetic mean will be distributed according to the normal distribution. During the experiment we only had one variable, the error rate. The error rate was randomly, uniform generated. The error rate did not depend on any other values of the observations.

We also observe that the arithemic mean of the runtime increases as our error rate increases, as shown in Figure 17. This is quite trivial since an increase in the error rate will cause more failures in the two phase commit, resulting in more delays. It is also worth mentioning, that the relationship between the error rate and the arithmetic mean of the runtime, seems to be exponential. The mean runtime should approach \infty as the error rate approaches \footnotesize 1.

Additionaly, it seems that the standard deviation is related to the error rate as well. As shown in Figure 18, we can observe that as the error rate increases, the standard deviation also does. A higher chance of failure will result in a bigger variation of results, assuming failures are uniformly distributed. We can also imagine it in a different approach. Let's imagine simple coin tosses. More coin tosses will result in a bigger possible variation of results. The relationship of the two variables seems to be exponential. The standard deviation should approach \infty as the error rate approaches \footnotesize 1.

Conclusions

From the observations we have seen that the expected runtime grows exponentially with respect to the error rate. Additionally, a growing standard deviation can also result in unpredictable results during runtime and also a worse real life worst-case scenario. Both bad average and worst case are quite undesirable. Thus we can agree that whilst designing a two phase commit implementation, the designer should have a low error rate (high reliability) "design goal" in a high priority. The costs of having low reliability are quite significant as explained above.

For completion, we want to mention, that polling the subordinates in case of a timeout, as assumed in this document, is only one way to deal with non-responding subordinates. But these are out of scope of our research.
